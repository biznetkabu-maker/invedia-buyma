# BUYMA 運用まとめ（保存版）

> このリポジトリの自動化ツールと、人がやるべき判断を一体で整理した**保存用ドキュメント**です。  
> 詳細手順・初回セットアップは `OPERATIONS.md`、商品取り込みの画面の流れは `INTAKE_MANUAL.md` を参照してください。

---

## 1. このプログラムができること・しないこと

| できること（自動・準自動） | しないこと（人の担当が中心） |
|---------------------------|------------------------------|
| 為替の取得（`forex.py` / `AUTO_FOREX` 等） | 「何を売るか」の**最初の発見・最終判断** |
| BUYMA 公開検索での**需要の目安**（お気に入り・競合・価格帯の抽出は環境により変動） | 真贋・仕入れ筋の**法的・倫理的担保** |
| 複数仕入先URLの**並列スクレイプ**と、条件に基づく**仕入れ先候補の比較**（`multi_source.py`） | `intake.py` への入力材料になる**商品の特定**（検索語・同一商品の確認） |
| 利益試算・**A〜E グレード**（`purchase_evaluator.py` 等） | BUYMA **出品フォームへの手動入力**（規約上、当面は手動出品を推奨） |
| 登録済みURLの**定期監視**・在庫・利益の再計算（`main.py`） | アカウント運用・表現・画像の**規約遵守の最終確認** |

**設計上の重要な分線**: 「検索結果ページから**商品URLを自動で確定する**」処理は壊れやすく、**検索URLの提示 → 人が商品ページURLを貼る**形に寄せています（`product_finder.py` の方針）。理由は、一覧ページのDOM変化が激しいこと、同一商品判定が難しいこと、日英表記ゆれで検索がズレること、ボット対策などです。

## 2. 主なコマンドの使い分け

| コマンド | 用途 |
|----------|------|
| `python3 intake.py` | **新規候補を1件ずつ**入力 → 需要の目安 → 仕入URL比較 → 判定 →（設定があれば）シート追記。**一問一答の対話**。 |
| `python3 intake.py --batch foo.csv` | スクレイプなしの**粗い一括グレード**。本登録の代替にはならない。 |
| `python3 main.py`（または GitHub Actions） | シートに載った商品の**監視・再計算**。「商品探し」用ではない。 |

---

## 3. `intake.py` の6ステップ（超要約）

1. **基本情報（手動）**: ブランド・商品名・カテゴリ・モデル年・**BUYMA予定売価（円）**・仕入通貨・為替（API＋確認）。  
2. **BUYMA需要（自動）**: 公開検索を開き、需要シグナルの**目安**を表示。  
3. **仕入先（半自動）**: **15サイト分の検索URL**表示 → ブラウザで同一商品の**商品ページURL**を貼付（複数はカンマ区切り可／空ならスキップ）。  
4. **スクレイプ・最安寄りの選定（自動）**: 在庫・利益率などに基づき候補を比較。URLなしなら現地価格の手入力。  
5. **判定（自動）**: A〜E、利益率、改善メモ。  
6. **シート（自動／D・Eは確認）**: 設定がなければ表示のみ。

詳細は **`INTAKE_MANUAL.md`**。

## 4. 人が商品を選ぶときの多角的な足場（保存メモ）

次の**5観点**で足切りし、残ったものだけ深掘りするとブレにくいです。

1. **需要**: なぜ今それが売れるかを**一言で説明**できるか。プラットフォーム内指標は**目安**。  
2. **経済（オールイン）**: 原価＋送料＋税・手数料・為替バッファ・トラブル余裕まで入れた後の利益。**利益率だけに依存しない**。  
3. **供給・物流**: 買える／届く／返品可否。ワンショット在庫はリスクが大きい。  
4. **コンプライアンス**: 仕入れ筋の説明、表現、画像、規約。**儲かりそう**より優先しがちなので明文化。  
5. **自分の運用**: 問い合わせ・トラブル対応・監視の時間。**同時点数を増やさない**ほど安全。

初心者向けの**型**としては、ガイドどおり**定番カテゴリ**（財布・バッグ・スニーカー等）× **説明しやすい需要**× **筋の通った仕入れ経路**から入るのが無難です（`beginner_brand_guide.md`）。

## 5. 資金の少ない初心者向け・運用最適化の核

- **プログラムは融資しない**。やることは**記録・監視・試算の型化**。  
- **同時在庫・同時案件を増やさない**（慣れるまで特に）。  
- **`intake.py` の D/E は止まれる人が強い**（確認ダイアログの意図どおり）。  
- **シート上の「追う件数」は少なければ少ないほど**、監視も mental 負荷も下がる。

## 6. 想定オペレーション定数（この保存版で置く前提）

**あくまで「週5分チェックリスト」用の例**です。自分の生活費・余力に合わせて**必ず書き換えてください**。

| 項目 | 保存版での想定例 | メモ |
|------|------------------|------|
| **同時在庫（抱える出品・仕入れ確定ベース）** | **最大 2 点** | 慣れないうちは **1 点** にしてもよい |
| **1点あたりの仕入れ上限（現地価格＋諸費用の目安）** | **¥80,000** | 例：この額を超える候補は `intake` 以前で却下してよい |
| **週あたりの新規検証** | **1 件まで** | 「追加の検討」はキューを作らない |
| **監視で追うシート行数の目安** | **5 行以内** | 超えたら「様子見」を整理してから追加 |

## 7. 週の作業チェックリスト（5分版）

**想定**: 同時在庫 **2 点まで**、1点上限 **¥80,000**、週の新規検証 **1 件まで**。

□ **（1分）在庫と上限の確認** … いま同時に抱えているのは **2 点以内**か。次に買うなら **¥80,000 以下**に収まる見込みか。  
□ **（1分）シートの顔ぶれ** … 監視対象は **5 行以内**か。**追わない候補**は行を残さずメモで区別する。  
□ **（1分）通知の有無** … 在庫切れ・停止・お知らせがあれば **BUYMA側の操作**とシートの整合。未対応を放置しない。  
□ **（1分）週1件ルール** … 今週まだ `intake.py` を **1 件も回していない**なら、**今日は回さない**と決めるか、**1 件だけ**回すかをはっきりさせる（両方やらない）。  
□ **（1分）次の7日の意思** … 新規を見るなら **ブランド＋商品名の検索語を1つ**決めてから起動する。**D/E は基本スキップ**。

**合計目安: 5分**。足りない場合は（1）（2）だけでも可。

## 10. 即買付モード（無在庫・注文後に仕入先で即決済）

**定義**: 出品時点では自前在庫を持たず、**BUYMAで注文が確定したあと、最短で**仕入先サイトで買い付ける運用。

### 10.1 有在庫モデルとの違い（運用の読み替え）

| 観点 | 有在庫寄り | 即買付（無在庫） |
|------|------------|------------------|
| シート上の「在庫」 | 手元・倉庫の有無 | **仕入先ページの在庫＝事実上の在庫**（間接在庫） |
| `main.py` 監視の意味 | 価格・利益・自前在庫に近い状態 | **いま売り続けてよいか**の目安。**欠品・`unknown` 連続**は「出品見直し・停止」のシグナルになりやすい |
| リスクの中心 | 寝かせた原資・保管 | **注文〜買付の時間差**での欠品・価格変動・先払いの積み上がり |
| 管理すべき上限 | 同時在庫点数 | **同時に抱える「買付前・未発送」案件数**と**月内先払い累計**（例として設定） |

### 10.2 最低限の運用ルール（プログラム変更なしで人側で固定）

- **注文検知〜仕入先で在庫・価格の再確認〜決済**までの**目標時間**（例: 数十分〜数時間）を決め、手順をテンプレ化する。  
- **注文時点の再計算**: 売価と買付総コストの差が許容を割ったら、事前に決めた**中止・顧客対応**に従う（現場判断の曖昧さが遅延を生む）。  
- **マージンは厚め**に寄せる（時間差リスクの保険）。ギリギリ利益のSKUは即買付では不利になりやすい。  
- **仕入URLが単一で不安定**なスクレイプ結果ばかりのSKUは、出品数を増やさない。

### 10.3 週次チェック（即買付向け・短縮）

**想定の置き方**: §6 の「同時在庫2点」は、即買付では **「同時オープン案件（買付前）最大2件」** のように**読み替えて**使ってよい（数値は自分で決める）。

□ **調達URLの健全性** … 監視で **欠品・`unknown` 多発**のSKUがない。あれば出品・価格の見直し。  
□ **オープン案件** … 買付前の注文が**上限以内**か。  
□ **先払いの累計** … 当月の見込みが**自分の許容を超えない**か。  
□ **`intake` / 試算の陳腐化** … 価格・為替の変動で**リスト済みSKUに地雷がない**か（必要なら手で再評価）。

## 11. 有在庫と無在庫で「同じプログラム」を使うべきか／最適解

**結論（現状の開発を変えなくてよい場合の最適解）**:  

- **同一のプログラム（`intake.py`・`main.py`・スクレイパー類）を使い続ける**のが妥当です。両者とも中核は同じであり、  
  「仕入れ先URLでの価格・在庫取得」「為替」「利益試算」「シートでの一元管理」です。  
- **変えるべきは主に運用ルールとスプレッドシート側の記録**です。たとえば行ごとに **「有／無在庫（即買付）」** をメモ列や運用記号で区別し、  
  **`main.py` の結果を有在庫は「自分の売価・在庫判断」に、無在庫は「調達ソースの死活」に**読み替える。

**なぜコードを複線化しないほうがよいか（無理に分叉しない理由）**:

- 監視・試算エンジンを二重にすると、**サイト変更時の修理コストが2倍**になりやすい。  
- 即買付と有在庫の差は、アルゴリズムより**意思決定とSLA・上限管理**が支配的です。

**いつコード側の分道を検討するか（将来の話）**:

- 「無在庫だけアラート閾値を別にしたい」「注文イベントとワークフロー連携したい」などが**常態化**したとき。**現時点で必須ではない**。  
- それまで **同一コード＋運用分担**がコスト対効果のバランスとして最適になりやすい。
